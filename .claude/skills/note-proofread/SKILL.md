# note-proofread

`note-pull` **직후**, 방금 내려받은 md 파일을 자연스러운 일본어로 다듬고 해시태그를 추천하는 스킬.
전제는 사용자가 pull한 파일 경로(`content/YYYY-MM-DD.md`)를 이 스킬과 함께 넘긴다는 것 — 그 파일 하나만 대상으로 한다.

핵심은 **git staging을 diff 리뷰 장치로 쓰는 것**이다. 원본(pull본)을 index에 고정한 뒤 working tree만
교정하면, `git diff`가 곧 "제안한 교정분"이 된다. 사용자는 `-`(원문)/`+`(교정)을 나란히 보며 검정한다.

> ⚠ 이 스킬은 **로컬 파일만 건드린다.** note로 되올리지 않는다. 교정 확정 후 반영은 사용자가
> 검토를 마치고 `note_push`로 직접 한다(CLAUDE.md의 단방향 push 원칙).

## 절차

1. **대상 파일 확인.** 사용자가 준 경로(없으면 방금 pull한 `content/YYYY-MM-DD.md`)를 Read.
   frontmatter가 있는 note md인지 확인한다. 아니면 사용자에게 되묻는다.

2. **원본을 stage에 고정.** 교정을 시작하기 전에 현재(=pull된) 내용을 index에 넣는다:
   ```bash
   git add content/YYYY-MM-DD.md
   ```
   - 이 stage된 버전이 diff의 비교 기준(원본)이 된다. **교정 편집보다 반드시 먼저** 실행할 것.
   - 파일이 untracked든 기존 추적 파일(재-pull)이든 상관없다. `git diff`는 늘 working↔index를 보여준다.
   - git repo가 아니거나 add가 실패하면, staging 없이 교정만 하고 3의 diff 대신 편집 요약으로 보고한다.

3. **본문만 교정.** frontmatter, 이미지(`![](...)`), 링크의 URL은 **그대로 둔다.** 본문 텍스트만:
   - **문법/조사/활용/오타를 자연스러운 일본어로.** 예: `同録`→`登録`, `飲みますた`→`飲みました`,
     `調べたには`→`調べたところ`, `なぜ〜なんだろうなら`→`なぜ〜なのかというと〜からです`.
   - **문장 끝에 가끔 적절한 이모지.** 모든 문장이 아니라 감정·소재가 뚜렷한 곳에만, 문맥에 맞게
     (예: 맥주🍺 / 영화🎬 / 책📚 / 아쉬움🥲). 원문에 이미 있는 이모지는 유지한다. 남발 금지.
   - **의미는 보존한다.** 문장을 지우거나 새 정보를 지어내지 않는다. 애매하면 원문 뉘앙스를 살린다.
   - 편집은 Edit로 문장 단위로 하거나, 본문을 통째로 Write해도 된다(frontmatter/이미지/링크 보존 필수).

4. **해시태그 추천.** 교정한 본문 내용을 근거로 note에 어울리는 해시태그를 5~8개 제안한다.
   - **일본어 태그**로, note에서 실제 쓰일 법한 자연스러운 것 (예: `#新宿` `#日記` `#名探偵コナン`
     `#東京生活` `#シェアハウス`). 지명·작품명·활동·감정/테마를 골고루.
   - frontmatter에 이미 있는 `tags:`(pull이 가져온 것)는 **유지**하고, 그 위에 **추가 후보**를 제안한다.
     중복은 빼고, 왜 그 태그인지 한 줄 근거를 곁들인다.
   - **자동으로 frontmatter에 밀어넣지 않는다.** 태그는 `note_push` 시 note로 반영되는 항목이므로,
     사용자가 고른 것만 `tags:`에 반영한다. "이 중 어떤 걸 넣을까요?"로 물어 승인받은 뒤 frontmatter를
     수정한다(수정분도 그대로 diff에 나타난다).

5. **diff 제시.** 교정 후:
   ```bash
   git diff content/YYYY-MM-DD.md
   ```
   staged(원본) vs working(교정본) = 제안 교정분. 사용자에게 주요 수정 항목(문법 교정 / 추가한 이모지 /
   반영한 해시태그)을 요약해 보고하고, IDE Source Control이나 위 diff로 검토하도록 안내한다.

6. **검토 후 선택지 안내.**
   - 일부만 되돌리기 → working tree에서 해당 줄만 원문으로 되돌림
   - 전부 원문으로 → `git checkout content/YYYY-MM-DD.md`
   - staging 해제(untracked 복귀) → `git restore --staged content/YYYY-MM-DD.md`
   - 확정 후 note 반영 → `note_push`

## 예시

입력: (note-pull로 `content/2026-07-04.md`를 받은 뒤) "이거 문법 자연스럽게 고치고 이모지 넣고 해시태그도 추천해줘"

```bash
git add content/2026-07-04.md            # 1. 원본(pull본) 고정
# ... 본문 교정 (Edit/Write) ...          # 2. 문법 + 이모지
# ... 해시태그 후보 제안 → 사용자 승인분만 frontmatter tags에 반영 ...
git diff content/2026-07-04.md           # 3. 제안 교정분 = 이 diff
```

→ `-同録`/`+登録`, `-飲みますた`/`+飲みました`, 문장 끝 🍺🎬📚 추가, frontmatter `tags:`에
승인된 해시태그(예: `#日記` `#東京生活`) 추가 등이 diff로 보인다.
사용자는 공부하며 검정하고, 확정분만 남긴 뒤 `note_push`로 되올린다.

## 실패 대응

- **대상 파일이 애매**(경로 여러 개/없음): 방금 pull한 파일이 무엇인지 사용자에게 확인.
- **git repo 아님 / `git add` 실패**: staging 없이 교정만 진행하고, diff 대신 "무엇을 어떻게 바꿨는지"
  편집 요약으로 대체 보고한다. (diff 리뷰 흐름은 포기하되 교정 자체는 완료)
- **이미 다른 변경이 stage됨**: pull 직후엔 드문 상황. 이 파일만 재-add해도 diff 비교엔 문제 없으나,
  다른 stage된 변경이 섞여 있으면 사용자에게 알린다.
